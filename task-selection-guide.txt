TASK SELECTION GUIDE

Purpose

Use this project to compare single-agent and multi-agent coding systems on issues
that can be turned into a real, testable software-fix task.

The goal is not "can the model talk about a fix?"
The goal is:
- real repo
- real code change
- real test command
- real pass/fail result


WHAT KIND OF REPOS TO PICK

Preferred repos
- Python repos
- repos with a small or medium codebase
- repos that install cleanly in a local sandbox
- repos with pytest or unittest already used
- repos where a narrow issue can be tested with 1 focused test file
- repos with stable, deterministic local test commands

Strong repo characteristics
- issue can be isolated to 1-3 files
- repo does not require external services to reproduce the bug
- repo does not require a database, browser farm, cloud credentials, or complex infra
- the repo has a clear local package layout and test layout
- failing behavior can be reproduced with a small regression test

Avoid repos like this
- heavy distributed systems
- repos that require Docker, Kubernetes, databases, or remote APIs
- front-end repos where success is mostly visual or subjective
- repos with slow or flaky tests
- repos with hard-to-install private dependencies
- repos where setup is harder than the bug itself


WHAT KIND OF ISSUES TO PICK

Pick issues with these traits
- one clear bug
- one clear expected behavior
- one clear pass/fail condition
- narrow scope
- objective validation
- can be expressed as a regression test

Best issue types
- environment variable behavior bugs
- CLI option parsing/default bugs
- plugin or loader selection bugs
- serializer/parser edge cases
- incorrect default handling
- missing guard condition
- wrong return value or wrong branch behavior
- formatting bugs ONLY if they can be checked by a deterministic test

Good issue shape
- bug is easy to describe in 1-3 sentences
- likely fix area is 1-3 files
- a targeted test can prove the issue is fixed
- no product judgment is needed to decide whether the result is correct

Avoid issues like this
- broad refactors
- feature requests that require new product design
- vague alignment or UI polish tasks without a deterministic test
- performance tasks without a stable benchmark harness
- issues that require subjective review instead of pass/fail validation
- issues that touch many unrelated modules
- issues where success depends on external systems


SIMPLE SCORING RUBRIC

Only pick issues that score well on this checklist.

Score 1 point for each "yes":
- Can I reproduce the bug with a focused test?
- Can I define a clear expected output?
- Can the likely fix stay within 1-3 files?
- Can the repo run locally in a sandbox?
- Can I evaluate success with a real test command?
- Is the issue a bug, not a redesign?

Interpretation
- 5-6 points: strong benchmark candidate
- 3-4 points: possible, but likely noisy
- 0-2 points: do not use


WHAT TO AVOID SPECIFICALLY FOR THIS REPO

This project works best when tasks use sandbox execution.

Avoid proposal-only tasks unless you have no better option.
If a task cannot be turned into:
- editable files
- test files
- test command
- sandbox workspace
then it is a weak benchmark task for this repo.

Bad benchmark pattern
- the model writes a "patch proposal" note
- no actual file changes are applied
- no real test is run
- validation becomes subjective

Good benchmark pattern
- the model returns structured edits or full file content
- the patch is applied in a sandbox copy
- the real test command runs
- the result is pass/fail


WHAT A GOOD TASK SPEC SHOULD CONTAIN

For each task, define:
- task id
- title
- repository
- short description
- constraints
- acceptance keywords
- validation instructions
- execution_mode = sandbox
- source_repo_path or fixture_dir
- editable_files
- readonly_files
- injected_test_files if needed
- test_command

If the issue is from a real repo, prefer adding an issue-specific regression test.


EXAMPLES OF GOOD TASKS FOR THIS PROJECT

Strong examples
- NO_COLOR support bug
- negative flag default bug
- plugin loader skips wrong class
- narrow option/default handling bug

Weak example unless upgraded with a real deterministic test
- vague alignment/rendering complaint without a reproducible sandbox test


HOW TO PREPARE A TASK

1. Pick a repo that fits the repo criteria above.
2. Pick an issue that fits the issue criteria above.
3. Find the likely editable file(s).
4. Create or inject a focused regression test.
5. Add the task in src/agentic_research/sample_tasks.py.
6. Prefer execution_mode = sandbox.
7. Make sure the test command can run locally and deterministically.
8. Run single-agent and multi-agent on the same snapshot.


HOW TO RUN IT

Run single-agent
PYTHONPATH=src python3 -m agentic_research.cli --architecture single --task TASK_ID --mode openai

Run multi-agent
PYTHONPATH=src python3 -m agentic_research.cli --architecture multi --task TASK_ID --mode openai

Run a benchmark suite
PYTHONPATH=src python3 -m agentic_research.benchmark_cli --tasks TASK_ID_1 TASK_ID_2 --architectures single multi --repeats 1 --mode openai


FINAL RULE

If the issue cannot reasonably be turned into:
- edit code
- run test
- get pass/fail

then do not use it for this benchmark.

Pick issues that let the benchmark measure implementation quality, not only reasoning quality.
